# PiLauncher 实例管理及核心功能闭环漏洞分析报告

本报告对 PiLauncher 当前的“游戏实例管理（Instance Management）”及围绕实例展开的核心功能（下载部署、备份恢复、模组管理、衣挂皮肤、云同步）进行了系统性梳理，深度分析了各链路中**未形成闭环（Open Loop / Broken Loop）**的漏洞与断点，并提出了相应的闭环优化建议。

## 1. 实例生命周期与部署修复闭环漏洞

实例生命周期管理主要涵盖：创建实例、下载部署核心及依赖、版本环境升降级、完整性校验以及启动链。

### 1.1 后端启动链缺乏“弱闭环自愈”机制
* **现象描述**：
  在启动游戏时，[PreLaunchCheckService](file:///h:/VSCodeWork/pilauncher/src-tauri/src/services/launcher/pre_launch_check.rs) 会对本地 JRE、库文件（Libraries）、原生库（Natives）及游戏资产（Assets）进行 SHA-1 校验。一旦检测到文件缺失或哈希不匹配，后端会直接抛出错误并强行中断启动链，由前端显示“修复弹窗”，引导用户确认并手动触发 `download_missing_runtimes` 进行修复。
* **未闭环分析**：
  这种“校验错误 -> 报错中断 -> 前端弹窗 -> 用户确认 -> 手动修复”的逻辑在用户交互层是闭合的，但在**纯后端代码层面属于非自愈的开环设计**。
* **带来的风险**：
  在一些微小的非关键依赖丢失（如仅仅缺失几个 KB 的本地 Native 或几张非核心贴图）时，阻断用户启动并强行弹出大范围的修复提示，会明显破坏玩家的启动流畅感。
* **改进方案（弱闭环自愈）**：
  后端应当引入自愈逻辑。在启动前检测时，若判定缺失的是**非大文件/非核心文件**，后端可以直接静默拉取修复，不阻塞当前的启动链；只有在缺失核心 JAR 包或大体量资产文件时，才阻断并返回前端引导用户。

### 1.2 下载引擎镜像源回退机制缺失
* **现象描述**：
  PiLauncher 底层强制或优先使用 **BMCLAPI** 等国内第三方镜像源来加速元数据、资产和第三方库的下载。
* **未闭环分析**：
  代码中对国内镜像源进行了硬编码或高优先级绑定，但**未设计自动的健康探测与官方源回退（Fallback）闭环**。
* **带来的风险**：
  当 BMCLAPI 发生服务宕机、限流或返回异常哈希值时，下载请求将持续报错并挂起，导致用户陷入“下载失败 - 无法游戏 - 无法自愈”的死胡同。
* **改进方案**：
  实现镜像源策略的闭环：引入网络探测层，当镜像源请求超时或报错次数达到阈值时，自动切换至 Mojang 官方源或其他备份镜像站（如 MCBBS 镜像），并记录错误以便后续退避重试。

### 1.3 部署取消后的垃圾清理开环
* **现象描述**：
  用户在下载部署实例或大型模组包的中途，可以点击“取消”中止任务。
* **未闭环分析**：
  在任务中断时，底层临时目录（`.tmp`）和已经下载的部分临时文件**没有被系统强制彻底清理**，状态也未做级联重置。
* **带来的风险**：
  长此以往，磁盘中会残留大量半成品的垃圾文件，占用用户空间。
* **改进方案**：
  在 `DownloadManager` 任务销毁或被取消的钩子函数中，增加事务性文件回滚清理逻辑，确保在任务宣告“已取消”的同时，物理垃圾文件也被全部清除。

## 2. 游戏备份与恢复闭环漏洞

存档备份功能旨在保护玩家的存档资产免受意外损坏。目前该模块在自动触发、安全监测及可恢复性上存在重大断点。

### 2.1 自动备份触发逻辑未落地（最严重的断点）
* **现象描述**：
  在 UI 界面和数据库中，都规划并展示了“退出游戏自动备份”（`auto_exit`）和“定时备份”（`auto_interval`）的策略配置。
* **未闭环分析**：
  在实际代码中，调用 [SaveManagerService::create_backup](file:///h:/VSCodeWork/pilauncher/src-tauri/src/services/instance/save_manager.rs) 的触发源只有两个：**手动创建备份**（写入 `manual` 标记）和**恢复前保护备份**（写入 `restore_guard` 标记）。当游戏进程退出并发出 `game-exit` 事件时，仅有日志服务 `useLogService.ts` 监听并处理了面板切换，没有任何逻辑去触发自动备份。
* **带来的风险**：
  对用户造成了误导。用户以为开启了“退出自动备份”，但实际上游戏崩溃或正常退出后，根本不会产生任何退出备份，极易导致玩家在存档损毁时无处回滚。
* **改进方案**：
  在后端启动管理中，在 `child.wait()` 侦测到进程结束并调用时长统计之后，触发 `SaveBackupScheduler`，根据实例设置自动将最新世界打包备份。

### 2.2 备份安全状态标记不可信
* **现象描述**：
  所有的存档备份记录中都包含 `safeBackup` 字段，且 UI 会据此显示“安全备份”绿色徽标。
* **未闭环分析**：
  代码中此标记由调用方硬编码直接传入 `true`，**系统并没有在备份时执行实际的安全性判定**。
* **带来的风险**：
  当用户在游戏运行期间强行通过后台命令备份，或者存档文件正处于高频写入状态、关键文件 `session.lock` 尚未释放时，直接压缩打包出来的文件可能是损坏的。但系统却依然将其打上“Safe”标记，诱导用户在未来恢复一个本身已经损坏的备份。
* **改进方案**：
  备份前进行文件占用及稳定性自检：
  1. 检查当前实例的游戏进程 PID 是否仍在运行。
  2. 探测存档目录的最后修改时间，确保有一段安全时间窗口（如 2 秒内无写盘变动）。
  3. 确认核心文件如 `level.dat` 可读且哈希稳定。不满足条件时，将 `safeBackup` 降级标记为 `false`，并在 UI 上向用户展示警告。

### 2.3 差异备份缺少恢复闭环
* **现象描述**：
  系统支持差异备份（仅打包自上一次全量备份以来修改时间更晚的文件）。
* **未闭环分析**：
  目前的实现方案只是做了一个“增量压缩包”，并不符合可恢复性闭环标准：
  1. **缺少删除清单**：如果在 base（全量备份）中存在某个文件，而在差异备份创建前该文件已被删除，恢复差异包时由于缺少删除动作记录，已删除的文件会被重新放回，导致存档冲突或污染。
  2. **允许删除 base 依赖**：系统允许用户在备份管理中随意删除底层的全量备份，而依赖该全量的差异备份不仅不会被级联清理，也未显示损坏警告。
  3. **base 缺失不阻断恢复**：在恢复差异包时，若底层 base 物理丢失，代码中未执行严格校验，可能只把干瘪的差异包解压出来，造成恢复出来的存档直接损毁。
* **改进方案**：
  1. 差异备份除了打包增量文件外，必须生成 `manifest.json`，在其中记录 `deleted` 文件清单，以便在恢复时物理删除已淘汰的文件。
  2. 删除备份时进行依赖校验：检测到被删除的全量备份仍有下属差异包依赖它时，阻止删除或提示用户级联删除。
  3. 恢复前校验依赖链条完整性，base 缺失则直接报错阻断。

### 2.4 恢复过程缺乏事务级回滚
* **现象描述**：
  当用户执行恢复操作时，代码逻辑为：直接将原存档目录物理删除，然后将临时解压的目标目录重命名移入。
* **未闭环分析**：
  此过程缺乏事务级隔离。一旦移入目标目录的步骤失败（如文件夹被外部占用、磁盘空间不足、权限问题），原存档已被物理清除，而新备份又未能恢复，导致存档双重丢失。
* **改进方案**：
  实施“双物理交换”回滚机制：
  1. 先将原存档重命名为临时回滚目录（如 `.rollback-<uuid>`）。
  2. 将新备份释放到目标路径。
  3. 校验目标路径文件完整性。如果失败，立即将原存档目录 `.rollback` 还原回来，并向用户报告故障；成功后，方可清理 `.rollback` 临时文件夹。

## 3. 模组集跨版本追踪与管理闭环漏洞

跨版本追踪用于帮助玩家判断一组 Mod（模组集）是否支持新的 Minecraft 版本及 Loader 环境。

### 3.1 追踪结果与实例自动更新开环
* **现象描述**：
  系统为用户提供了“目标环境追踪”功能，能检测出某个模组集（Mod Set）在目标环境（如 `1.21.2 + Fabric`）下的兼容情况，并计算出就绪比例（如 8/12 已就绪）。
* **未闭环分析**：
  该功能目前止步于**“信息展示”**阶段。当追踪结果达到 100% 且用户想一键应用到当前实例时，系统没有提供打通当前实例的自动升级/替换入口，只停留在“提示就绪”的开环状态。
* **带来的风险**：
  玩家无法方便地一键将实例升级并装配好追踪到的新版本模组，只能将列表截图，再逐个去实例下载页面下载，使用效率大打折扣。
* **改进方案**：
  打通“追踪 -> 升级实例 -> 部署”的完整闭环。当追踪就绪后，提供“一键升级当前实例至此追踪版本”功能，自动修改实例环境，调用下载管线把追踪到的新版 jar 包批量替换进 `.minecraft/mods` 目录。

### 3.2 模组集删除时的孤儿追踪器漏洞
* **现象描述**：
  在 [LibraryPage.tsx](file:///h:/VSCodeWork/pilauncher/src/pages/LibraryPage.tsx) 中，仅在前端点击删除模组集按钮时，才会手动触发 `removeTrackersForCollection`。
* **未闭环分析**：
  后端 `remove_collection` 接口在数据库层面只级联清理了 `collection_items`。如果 Collection 实例因其他后端命令或异常流被直接删除，对应的 `mod_set_trackers` 和 `mod_set_tracker_items` 表记录不会级联清除，成为遗留在数据库里的僵尸数据。
* **改进方案**：
  在 SQLite 数据库层面，将 `mod_set_trackers` 和 `mod_set_tracker_items` 通过外键 `ON DELETE CASCADE` 约束绑定到 `collections` 表；或者在后端 Rust 的 `remove_collection` 服务级逻辑中，强制加入同步清理追踪器的事务代码。

### 3.3 数据库字段与前端 UI 交互开环
* **现象描述**：
  数据库中对 `collections` 预留了 `position` 字段用于排序，并预留了 `cover_image` 用于定制封面。
* **未闭环分析**：
  前端 UI 层完全**缺失拖拽排序组件**，这导致 `position` 虽然在每次插入新模组时被默默累加，但始终无法被用户主动改变。同时，由于完全没有设计封面图上传和编辑的入口，`cover_image` 字段沦为占位符。
* **改进方案**：
  在 `Library` 页面引入轻量拖拽库，支持用户自定义排序，并在修改位置后触发批量 `update_collection_item_positions` 接口；增加模组集元数据编辑弹窗，允许用户上传本地图片以丰富模组集/整合包的封面显示。

## 4. 衣挂皮肤系统闭环漏洞

衣挂（Wardrobe）页面用于管理游戏内的皮肤和披风，支持本地资产的管理与微软/第三方账号的多设备同步。

### 4.1 模态窗退出后的焦点重置 Bug（致命 UX 开环）
* **现状分析**：
  键盘或手柄用户在滚动浏览庞大的皮肤库时（例如焦点落在第 20 张皮肤上），点击 Enter 打开操作菜单模态窗（`WardrobeSkinMenuModal`）。当执行应用或者直接按下取消关闭模态窗后，该模态窗的 DOM 被卸载。
* **未闭环分析**：
  系统在模态窗关闭后，**没有主动将空间导航焦点还给触发模态窗的那个卡片 Key**。焦点重置机制会自动退避并强行把焦点扔回页面顶部的 `'wardrobe-upload-card'` 或 `'wardrobe-skin-0'`。
* **带来的风险**：
  这会导致滚动条瞬间被强行拉回到页面最上方，用户不得不重新痛苦地向下滚屏搜寻之前的浏览位置，焦点管理链条在此处发生致命断裂。
* **改进方案**：
  在 React 组件中设计焦点记忆闭环：打开模态窗前，使用 Ref 记录当前被聚焦的 DOM 标识（如 `lastFocusedKey.current = activeFocusedKey`），当模态窗关闭后，立即调用 `setFocus(lastFocusedKey.current)`，令滚动条和焦点纹丝不动地回到原位。

### 4.2 微软资产同步时皮肤面板无状态反馈
* **现象描述**：
  点击刷新时，系统会在后台通过 WebDAV / 微软接口同步最新的皮肤资产（`hydrateWardrobe`）。
* **未闭环分析**：
  同步进行中，披风面板 `WardrobeCapePanel` 会响应 `isLoadingProfile` 状态并正确展示骨架屏（Skeleton Card）；但皮肤面板 `WardrobeSkinPanel` 却**完全忽略了该状态**，在加载过程中皮肤列表没有任何 Loading 指示器或淡色遮罩。
* **带来的风险**：
  在网络传输较慢时，用户会看到一个完全静止的列表，无法判断同步是已经完成还是正在读取中，可能导致重复多次点击刷新，造成接口压力和数据混乱。
* **改进方案**：
  将皮肤面板的数据渲染逻辑与 `isLoadingProfile` 深度绑定，当加载状态为 `true` 且本地数据正在更新时，渲染出带占位动画的骨架卡片（Skeleton Card）。

### 4.3 提示字符Mojibake乱码导致报错闭环失败
* **现象描述**：
  [WardrobeCapePanel.tsx](file:///h:/VSCodeWork/pilauncher/src/features/wardrobe/components/WardrobeCapePanel.tsx) 的多处错误及限制提示字符串（如微软正版账号限制、无披风提醒）硬编码在文件中，且在编译/转换时发生了严重的 Mojibake 乱码（例如出现 `鎶鍒囨崲闇€瑕佷...` 等乱码字样）。
* **未闭环分析**：
  乱码使得提示完全丧失了可读性，这在异常反馈循环（Error feedback loop）中形成了死结，用户在发生异常时根本看不懂系统在说什么。
* **改进方案**：
  移除该文件中所有的硬编码文本，将其迁移至国际化 json 包（如 `zh-CN.json`）并进行编码统一（确保全部为 UTF-8 格式），用 `t('wardrobe.cape.microsoftRequired')` 等提取翻译文字进行干净利落的渲染。

## 5. WebDAV 同步与配置共享闭环漏洞

同步系统依托追加式操作日志（Append-only Operation Log）和 LWW（Last Write Wins）策略设计，支持本地收藏和数据的自动云同步。

### 5.1 实例级配置及多人游戏绑定未纳入云同步
* **现象描述**：
  目前的 WebDAV 同步服务（[WebDavSyncService](file:///h:/VSCodeWork/pilauncher/src-tauri/src/services/webdav_sync_service/mod.rs)）仅支持 Favorite 收藏项、衣挂皮肤资产以及存档备份的同步。
* **未闭环分析**：
  实例自身的配置（Instance Config，如绑定的多人游戏服务器、最大分配内存、是否自动连接服务器、自定义 JVM 启动参数）**完全保存在本地 SQLite 数据库或本地 json 文件中，没有被纳入 WebDAV 的同步覆盖范围**。
* **带来的风险**：
  多端一致性漏洞。如果用户在 PC 上配置好了某个实例的复杂 Java 调优参数并绑定了局域网服务器，当他在另一台设备（如 Steam Deck）上拉取该实例时，所有绑定的服务器、内存分配参数都需要重新手动配置。
* **改进方案**：
  在追加式操作日志中，支持并定义 `instance_config_update` 类型的操作，当用户更改实例设置时，向同步序列中加入该操作，使实例参数也能享受秒级云同步。

### 5.2 冲突解决与异常处理人机交互闭环缺失
* **现象描述**：
  同步引擎在后台运行，或者在启动/退出应用时自动触发同步。
* **未闭环分析**：
  当 WebDAV 遇到异常（如密码过期、网络代理不可达、文件锁冲突超时等），**后端仅会向前端控制台抛出一个底层字符串错误，缺失人机交互的引导与回滚确认机制**。
* **带来的风险**：
  如果发生同步错误，用户既无法感知当前数据处于“未同步”还是“同步被锁住”状态，也无法在界面上方便地重新输入密码、强制清除 WebDAV 上的过期锁（Lock Token）或者强制选择“以本地为准/以云端为准”。
* **改进方案**：
  在设置的“数据”栏或资源库顶部，增加一个 WebDAV 同步状态中心挂件：
  1. 可视化当前同步状态（已同步 / 同步中 / 异常中断）。
  2. 发生错误时，弹出友好的向导，允许用户快速测试连接、查看详细冲突原因并一键选择“清除云端锁”、“覆盖云端”或“合并拉取”。

## 6. 总结与演进路线图

当前 PiLauncher 在基本的用户界面功能展示上已经非常丰满，但是当涉及到**数据持久化、异常恢复、全键盘焦点管理以及多设备数据流动**时，暴露出多处“预留概念未落地”和“交互流中途断裂”的问题。

为实现极致的 OreUI 精品标准，建议后续的演进路线规划如下：

```
P0 (核心可用性闭环)
 ├── 1. 修复 Wardrobe 模态窗关闭后焦点飘逸与页面强制置顶的 Bug (P0.1)
 ├── 2. 汉化并国际化 WardrobeCapePanel 中的 Mojibake 乱码 (P0.2)
 └── 3. 后端接入 game-exit 事件，打通退出自动备份 (auto_exit) 的完整触发管线 (P0.3)

P1 (安全性与事务闭环)
 ├── 1. 差异备份补充 manifest 记录以及 deleted 文件清单支持 (P1.1)
 ├── 2. 实现存档恢复时的 .rollback 事务保护，防止文件写入失败导致存档双失 (P1.2)
 └── 3. 限制备份被依赖的 Base 全量包删除，防止孤儿差异包的产生 (P1.3)

P2 (体验与一致性优化)
 ├── 1. 增强 WebDAV 冲突和文件锁错误在前端的弹窗指引，完成异常自愈闭环 (P2.1)
 ├── 2. 实例级的多人游戏服务器绑定与 Java 配置参数纳入云同步操作日志 (P2.2)
 └── 3. 模组集 Collection 被删除时，后端在数据库层面级联删除关联的 Trackers (P2.3)
```

通过分步将这些功能从“能用”提升至“高容错、自愈、全端对齐”，PiLauncher 将拥有极佳的健壮性。
